# 一、WHY —— 为什么世界需要弈

## 1.1 灵感与溯源

弈来自三条线的交汇:

1. **围棋的协作哲学** —— 人"布势"(宏观战略),棋子"落子"(微观执行)。势由心生,子随势动。这天然对应 AI Agent 时代最理想的范式:用户说"要什么",Agent 想"怎么做"。"弈"是围棋古称,既是产品名,也是哲学宣言。
2. **Flowith 的知识花园/市场** —— Flowith 证明了 AI 时代的知识可被封装、上架、交易:Seed 系统把文件拆成最小知识单元,买一个知识库相当于给大模型装"知识外挂"。这让弈意识到:社区不只分享"怎么做"(工作流),更能分享"知道什么"(知识库)。
3. **开源 Agent 的格局盲区** —— 开源 Agent 不缺(OpenCode / Goose / Hermes / OpenClaw),但都扎堆在"编码"一件事上:CLI/TUI 优先、面向开发者、围着代码库转。"通用桌面自动化 + 面向非编码的知识工作者 + 带知识市场飞轮"的开源 Agent,是空白。弈不打"唯一开源"这张牌(已不成立),打"场景跨界 + 生态飞轮"。

## 1.2 痛点与问题

**个人用户:**
- 现有 AI 助手是"工具人"——一步步喂指令,不会规划、不会记住、不会从历史任务学习
- 重复任务每次从头描述,Agent 没记忆、没"棋谱"
- 敏感数据(财务、代码、私人文档)不敢交给云端 Agent

**知识工作者:**
- 专业知识锁在专家脑里或散落文档,无法被 AI 复用
- 整理知识耗时却无变现渠道——行业报告、投资笔记、技术文档只能免费分享或烂在硬盘

**开源社区:**
- 想贡献 Agent 能力的人没有低门槛入口——要么写核心代码,要么什么都不做
- 没有"经验"级别的贡献:高手解题路径无法沉淀、复用、传播

## 1.3 用户与场景

| 用户 | 弈中称谓 | 画像 | 核心诉求 |
|------|----------|------|----------|
| 普通用户 | **执棋者** | 知识工作者、效率党、不写代码的人 | 自然语言指挥 Agent 整理文件/写周报/爬数据/批量处理。要"省心" |
| 创作者 | **布道者** | 行业专家、资深开发者、内容创作者 | 把专业知识/工作流封装成手筋或棋经上架。要"变现" |
| 贡献者 | **弈匠** | 开源开发者、AI 工程师 | 扩展插件、改进核心、贡献能力棋子。要"掌控感"与"影响力" |

**一个故事讲完三类用户:**
投资分析师(执棋者)按 `Cmd/Ctrl+Shift+Y` 说"把这三份研报摘要成投资备忘录,格式参考我上周那份"。弈从本地棋经取上周备忘录做模板,调用社区"研报摘要"手筋,3 分钟生成。他一键发布这套流程到弈林,标价 $1.99。新手买下,站在他肩膀上完成同样的事。一位弈匠为这个手筋贡献了更好的 PDF 解析能力棋子,获得社区声望。

## 1.4 市场与定位

弈所处的交叉地带(无人占据):

| 已有玩家 | 占据什么 | 弈的差位 |
|---|---|---|
| OpenCode / Goose / Hermes / OpenClaw | 开源编码 Agent,开发者向,CLI/TUI | 弈做**通用桌面**(文件/浏览器/终端/多模态),面向**知识工作者**,GUI 优先 |
| Raycast / Alfred / AutoHotkey | 桌面快捷指令,无 AI 规划、无记忆 | 弈有规划、有棋谱、有棋经 |
| AutoGPT / Devin / Cursor | 强 Agent,但云端或绑死编码场景 | 弈本地优先 + 通用桌面 + 开源 |
| Obsidian / Notion / Logseq | 本地知识库,知识是"死"的 | 弈的棋经是 Agent 的外挂大脑,可调用可变现 |
| Flowith | 有知识市场,但闭源 + 云端 + 非桌面 | 弈用开源 + 桌面 + 本地优先,切它够不到的隐私敏感/离线/自托管市场 |

**定位句:**
> 开源的桌面智能体——你说意图,它落子执行;你沉淀知识,它越用越聪明;你分享智慧,它帮你变现。

## 1.5 差异化壁垒(四道,由浅入深)

1. **哲学壁垒** —— "布势-落子"交互范式 + 围棋隐喻贯穿全产品。功能可抄,一以贯之的语言体系和气质抄不走。是社区认同感的地基。
2. **通用桌面 + 面向非开发者** —— 编码 Agent 群里没人认真做"非编码用户也能用"的桌面 Agent。GUI 优先、圈选即语境、全局唤醒、语音输入——这些是把 Agent 从开发者工具推向大众的工程活,不是壁垒本身;壁垒是"第一个做对的 + 用围棋隐喻把用户教育成'执棋者'"。后来者要抄,得同时抄交互 + 社区语言 + 棋谱格式,光抄代码没用。
3. **棋谱(.yi.json)作为开放标准** —— 每次任务的结构化执行记录,可分享、可复现、可交易。若"棋谱"成为桌面 Agent 领域的"MP3",弈就是格式定义者。
4. **弈林生态飞轮(最深)** —— 优质棋经/专家包吸引用户 → 用户带来创作者 → 创作者产出更多优质内容 → 平台价值上升。Flowith 已验证此模式,弈用开源 + 桌面 + 本地优先切入它够不到的市场。飞轮转起来,后来者难追。

# 二、WHAT —— 弈是什么

## 2.1 产品概念

弈是一个开源桌面智能体,核心是**一套"资产协议 + 流通市场 + 组装引擎"**:
- Agent 是引擎
- 资产是燃料
- 弈林是集市

用户布势(说意图)、Agent 落子(执行)、复盘沉淀为新资产、弈林让资产流通,组装起来就是一个"专家"。

## 2.2 资产本体论(弈的底层模型)

弈的世界里流通六类资产:
- **四类通用零件**(业界已有,弈采纳并标准化):手筋、器、驿、棋魂
- **一类弈独有零件**:棋经
- **一类组合体**(弈的创新):专家包——把零件组装成可流通的"专家"

壁垒不在通用零件本身,在棋经 + 专家包 + 它们的流通协议。

### 2.2.1 通用资产(采纳标准,非壁垒)

| 资产 | 弈中称谓 | 本质 | 类比 |
|---|---|---|---|
| Skill | **手筋** | 可复用的能力单元(一段 prompt + 工具调用逻辑) | 当下 Agent 生态的 skill |
| Plugin | **器** | 扩展 Agent 工具面的可插拔模块 | 通用 plugin 概念 |
| MCP | **驿** | 连接外部工具/数据源的协议接入 | Model Context Protocol |
| Character | **棋魂** | 人格卡:定义 Agent 的语气、思维风格、价值观,类似 `soul.md` | AI persona / soul |

这四类是"零件",业界已有,弈不重新发明,只做**统一格式 + 统一市场入口**。壁垒不在零件本身。

### 2.2.2 弈独有资产:棋经(Qijing)

棋经是一个**结构化的专项知识库**,可近似理解为一个"专项 LLM wiki":不是散乱文档,是被组织成可被 Agent 检索、引用、推理的知识结构。

**为什么这是弈的独有物,而不是又一个 RAG:**
- Obsidian/Notion 的知识是**给人读的**;弈的棋经是**给 Agent 用的**——结构为检索和推理优化,不为浏览优化。
- 普通 RAG 是**一次性查询**;棋经是**可命名、可引用、可组合、可上架、可版本化**的持久资产。用户能说"按我那个'投资备忘录'棋经的格式",Agent 就懂。

**棋经的两层结构:**

| 层 | 弈中称谓 | 本质 | 类比 |
|---|---|---|---|
| 原文层 | **典籍 (Source)** | 原始资料:PDF、笔记、报告,逐字保留,版本化 | 学者的原典 |
| 精炼层 | **简编 (Digest)** | AI 把典籍拆成的最小知识单元,结构化、可检索、可引用 | Flowith 的 Seed |

**分工:** Agent 检索用**简编**(快、准、省 token),引用溯源回**典籍**(权威、可验证、不丢原文)。用户买一个棋经,两层都拿到——简编用来让 Agent"会",典籍用来让自己"信"。

**已定方向:** 结构化、可检索、可引用、可版本化、可交易。**具体 schema 留待专题。**

### 2.2.3 资产的组合:专家包(Expert Pack)

资产单独是零件,组合起来就是"专家":

```
棋经(知道什么) + 棋魂(怎么想) + 手筋(能做什么) + 器/驿(用什么做)
                          ↓
                    = 一个有专业知识的专家
```

**例子:** "顶级投资人心法"专家 = [投资偏好棋经] + [冷静分析型棋魂] + [研报摘要手筋 + 财报解读手筋] + [浏览器插件 + SEC API MCP]。

用户可以:
- 自己组装专家(从市场挑零件)
- 直接在市场买/换别人组装好的**专家包**(打包好的资产组合)
- 把自己调教好的专家包上架流通

**这一步把"资产"从静态零件升维成"可流通的能力体"。** 这是弈林飞轮的真正引擎:零件市场 + 专家包市场,两层流通。

## 2.3 交互四模块(资产的生产与消费面)

四个模块本质是"用户如何创造、调用、沉淀、流通资产":

| 模块 | 弈中称谓 | 做什么 | 与资产的关系 |
|---|---|---|---|
| 布局 | **布局** | 自然语言/语音/圈选/挂角定式 → 表达意图 | **消费**:调用专家包/手筋/棋经 |
| 落子 | **落子** | 推演→分投→执行→收官,可视化 | **执行**:激活资产完成任务 |
| 复盘 | **复盘** | 自动生成棋谱(.yi.json)、个人棋经、双向链接、版本控制 | **生产**:把这次执行沉淀为新资产(棋谱/棋经) |
| 弈林 | **弈林** | 市场:分享/交易 | **流通**:资产与专家包的集市 |

**一句话闭环:** 布局消费资产 → 落子执行 → 复盘生产新资产 → 弈林让资产流通 → 回到布局。飞轮自转。

## 2.4 弈不是什么(边界)

- 不是 ChatGPT 客户端(真实操控桌面,不只聊天)
- 不是编码专用 Agent(通用桌面,编码是众多场景之一)
- 不是纯笔记软件(棋经是 Agent 外挂大脑,不是人读的笔记本)
- 不是 Flowith 的克隆(开源 + 桌面 + 本地优先,切它够不到的市场)

# 三、HOW —— 弈怎么运转

## 3.1 交互范式深挖:布势-落子

### 3.1.1 布势(用户如何表达意图)——三重境界

| 境界 | 弈中称谓 | 机制 | 背后消费的资产 |
|---|---|---|---|
| 口述大势 | **言** | 全局快捷键(`Cmd/Ctrl+Shift+Y`)唤出极简输入条;文本/语音(本地 Whisper)/文件拖拽混输 | 无,纯自然语言意图 |
| 拈子示意 | **示** | 桌面圈选文件 / 网页高亮文字 / 截图区域 → 右键"弈" → 自动进上下文 | 无,上下文即语境 |
| 挂角定式 | **式** | 一键调用专家包/手筋(从市场或本地) | 消费:专家包/手筋 |

**布势的产物:** 一个"意图 + 上下文 + 可选挂载资产"的组合,传给落子引擎。

### 3.1.2 落子(Agent 如何执行)——四步可见

| 步 | 弈中称谓 | 做什么 | 可视化 |
|---|---|---|---|
| 1 | **长考(推演)** | 展示推理链:"我打算 1.…2.…3.…" | 半透明"小目(思考窗)"滚推理链 |
| 2 | **分投(拆解)** | 大任务拆子任务 | 棋枰上暗色虚影棋子预排布 |
| 3 | **落子(执行)** | 真实操控桌面:读写文件/调 API/开浏览器/跑终端。不模拟 | 实心光效棋子逐枚点亮 |
| 4 | **收官(总结)** | 生成改动摘要/耗时/资源/关键决策 | 金色棋子(总结卡) |

**关键:真实操作 + 可干预。** 落子过程中用户可随时**打挂(暂停)**插入修正,Agent **悔棋(回滚)**到任意前序步骤(改动备份至 `.yi_undo`),用新指令重落。

### 3.1.3 复盘(如何沉淀资产)

| 机制 | 产出 | 资产类型 |
|---|---|---|
| 自动生成**棋谱** | `.yi.json`:输入→推理→执行→结果全流程 | 棋谱(执行记录) |
| 棋谱检索 | 新任务时自动检索历史相似棋谱复用 | 棋谱复用 |
| 个人棋经 | 用户笔记/文档/棋谱在本地构成个人知识库,Agent 可引用 | 棋经(典籍+简编) |
| 双向链接 | `[[]]` 语法连知识节点,网状结构 | 棋经内部结构 |
| 版本控制 | 基于 Git 存储 | 所有资产 |

### 3.1.4 弈林(如何流通)

见 3.2 市场模型。

## 3.2 市场模型

### 3.2.1 三种流通方式

| 方式 | 含义 | 例子 |
|---|---|---|
| **散件流通** | 每类资产独立上架:手筋/器/驿/棋魂/棋经 各自标价或免费 | 上传一个"研报摘要"手筋,$0.99 |
| **整包流通** | 把多件自有资产打包成专家包,整体上架 | [自研投资棋经 + 自定棋魂 + 两个手筋] 打包成"价值投资者"专家包 |
| **引用组包** | 包里**引用**别人的棋经 + 自己的棋魂/手筋,不复制原作 | "我的财报分析专家"包:引用 @A 的"SEC 文档棋经" + 自己的"冷静型棋魂" + 自研手筋。买家拿到包,运行时按引用拉取 @A 的棋经 |

### 3.2.2 引用模型的关键性质(弈市场的灵魂)

- **不复制,只引用** —— 被引用的棋经仍归原作者,版本/价格/下架都由原作者控制。包作者只是声明依赖。
- **收益沿引用链分账** —— 买家付包价,系统按包作者设定的分账比例(或默认比例)自动拆给被引用棋经的原作者。像 npm,但带分账。
- **依赖图可追溯** —— 任何一个专家包都能展开看它引用了谁的什么棋经。attribution 不丢,声望和钱都不丢。
- **原作者随时撤引用** —— 原作者下架/涨价,所有引用它的包收到通知;买家已购的版本冻结,新买家按新条件。

### 3.2.3 两种交易模式

| 模式 | 说明 |
|---|---|
| **分享 (Share)** | 免费,任何人可取,记声望 |
| **交易 (Trade)** | 明码标价,付费购买 |

### 3.2.4 资产在市场的分层

```
零件市场(散件)        ← 手筋 / 器 / 驿 / 棋魂 / 棋经 各自流通
        ↓ 组合
专家包市场(整包)      ← 自有件打包 / 引用他人件 + 自有件 组包
        ↓ 飞轮
弈林生态              ← 创作→流通→分账→再创作
```

**一句话:** 弈林是带分账的 npm + 带引用链的 GitHub Fork + 带变现的知识市场,三层叠在同一个资产协议上。

## 3.3 商业模式

弈是开源项目,但开源 ≠ 不赚钱。三层收入:

| 层 | 来源 | 性质 |
|---|---|---|
| **市场抽成** | 弈林交易的手续费(默认 10%,可调) | 主收入。分享免费、交易抽成 |
| **官方专家包** | 弈团队自己出品的高质量专家包/棋经销售 | 次收入,兼做质量标杆 |
| **托管/增值服务**(远期) | 云端模型代理、托管棋经同步、团队协作版 | 可选收入,不强制 |

**开源 + 市场的分界:** 核心引擎(布势/落子/复盘/市场协议)永远开源免费;**内容**(棋经/专家包)由创作者定价,弈只抽成。代码免费,知识付费——和 npm 公开包 + 私有付费包一个逻辑。

## 3.4 生态飞轮(自转条件)

```
优质棋经/专家包 → 吸引用户
        ↓
用户增多 → 创作者收益上升 → 更多创作者入场
        ↓
资产更丰富更优质 → 平台价值上升 → 吸引更多用户
```

**冷启动关键:** 弈团队先自产 3-5 个高质量官方棋经/专家包做种子(如"研报摘要""代码 review""周报生成"),证明模式跑得通,再开社区贡献。

## 3.5 路线图(产品阶段,非技术)

| 阶段 | 目标 | 标志 |
|---|---|---|
| **MVP(0→1)** | 单机跑通"布势-落子-复盘"闭环,无市场 | 一个用户能用自然语言指挥 Agent 完成真实桌面任务,棋谱存得下、能复用 |
| **种子期(1→N)** | 加上弈林市场(只分享,不交易) | 社区能上传/下载手筋和棋经,出现非团队贡献者 |
| **商业化(N→飞轮)** | 开启交易 + 引用组包分账 | 第一笔付费交易发生,创作者拿到分成,飞轮启动 |
| **生态期** | 官方专家包矩阵 + 创作者经济成形 | Top 创作者月收入可观,弈成为"AI 时代知识创作者的 OnlyFans" |
